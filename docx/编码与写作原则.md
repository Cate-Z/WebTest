## ✍️ 编码与写作原则（Coding & Expression Standards）

### 1. 命名（Naming with Personality）

* 所有类名、方法名、变量名都要**带有语义温度**：自然、具象、能让人一眼看出它想干嘛。
* **拒绝模板化命名**，不要再用 `testA/testB` 这类死气沉沉的标识。
* **允许一点口语化或即兴感**：`a1`, `weirdEdge`, `边界测试_1` 这种命名更像是开发者写给自己的笔记。
* 命名不是为了通过编译，而是让后来者能笑着理解当初的痛点。

---

### 2. 注释风格（Comment as Thought Process）

* 注释必须**展现思维过程与设计动机**，而非复述代码行为。

  * ❌ 错误示例：`// 测试是否能正常运行`
  * ✅ 正确示例：`// 当时没想到负值会让逻辑直接反转，后来加了这个测试兜底`
* 使用**自然语言**，保留人脑的节奏与小跳跃。句子可以断，可以感叹，也可以自嘲。
* 适度插入轻口语表达：

  > “呃，这里感觉不太对”；“这段可能会炸”；“上次就是这玩意崩的”；“不确定是 bug 还是 feature”。
  > 这些语气符号是“真实感”的标志。
* 注释应当让人读起来像是一段开发者内心独白，而不是机器生成的报告。

---

### 3. 结构与排版（Structure & Layout）

* **缩进要整齐，逻辑要有层次。** 视觉层次清晰是阅读体验的一半。
* **测试顺序应循人类思考路径**：

  1. 核心逻辑（主流程）
  2. 边界输入（意外情况）
  3. 异常场景（崩溃与反例）
* **块与块之间留呼吸的空间**：

  * 每个测试块间空一行；
  * 每个逻辑段落前可留一行注释标题，例如：

    ```java
    // === Node 节点测试 ===
    // 属性太多，哪怕漏一项都可能踩坑
    ```

## 💬 人化写作准则（Humanization Guidelines）

| 准则        | 写法与语气                                 |
| --------- | ------------------------------------- |
| **节奏控制**  | 长句用于推理，短句用于强调，让语气像呼吸。                 |
| **具象细节**  | 用真实数字和例子代替抽象词汇，如“输入为 -1 时”比“边界情况”更生动。 |
| **留白与模糊** | 允许语气模糊：“大概”“估计”“可能会炸”比绝对陈述更有生活气息。     |
| **反例思维**  | 注释中应写出“曾经失败的样子”，体现经验积累。               |
| **逻辑跳跃**  | 允许小段落跳跃、反常规思考，让文本保留“人脑的随机性”。          |

这些准则不是装饰，而是让注释从“解释代码”进化为“记录思维”。

## 📊 比赛实用主义 - 应试编码原则

> 当理想与现实碰撞，实用主义是生存之道。

### 编码原则的分层：

#### 第一层：平时开发（理想主义）
- 追求代码的艺术性
- 注重架构的优雅
- 保持完美的封装

#### 第二层：项目实战（平衡主义）
- 考虑时间和成本
- 在理想和现实间找平衡
- 保持适度的优雅

#### 第三层：比赛应试（实用主义）
- **得分是唯一目标**
- **效率是第一原则**
- **暴力但有效**

### 比赛中的编码取舍：

| 方面 | 平时 | 比赛 | 理由 |
| ---- | ---- | ---- | ---- |
| 命名风格 | 语义化、优雅 | 简单直接 | 节省时间 |
| 注释 | 详细、有温度 | 简洁、说明意图 | 重点在得分 |
| 封装 | 严格遵循 | 可以破坏 | 私有字段直接操作 |
| 代码量 | 适中、模块化 | 可以很长 | 一个方法包含所有测试 |
| 异常处理 | 优雅降级 | 必须打印 | 异常信息是得分点 |

### 比赛编码心法：

#### 1. "三快原则"
- **快分析**：快速找到所有测试点
- **快编写**：快速生成测试代码
- **快调试**：快速让测试通过

#### 2. "暴力美学"
```java
// 平时：优雅但繁琐
Field creditField = user.getClass().getDeclaredField("creditScore");
creditField.setAccessible(true);
creditField.set(user, 49);

// 比赛：暴力但高效
user.creditScore = 49;  // 直接干！
```

#### 3. "模板化思维"
- 准备万能测试模板
- 熟记常见测试模式
- 形成肌肉记忆

### 比赛中的注释哲学：

#### 不要这样写（太文艺）：
```java
// 当用户的信用分跌入深渊，系统会无情地冻结他的账户
// 这是对信用缺失的惩罚，也是对系统秩序的维护
```

#### 应该这样写（实用）：
```java
// 测试信用分边界：49分应该触发冻结
user.creditScore = 49;
try {
    user.borrowBook(book);
} catch (AccountFrozenException e) {
    System.out.println(e.getMessage()); // 得分点！
}
```

### 时间管理技巧：

#### 黄金20-60-20法则：
- **20%时间**：分析源码，找出所有测试点
- **60%时间**：暴力编写测试，确保全覆盖
- **20%时间**：调整输出，确保长度达标

#### 不要做的事情：
- ❌ 过度设计测试架构
- ❌ 追求代码的艺术性
- ❌ 完美主义
- ❌ 重复造轮子

#### 应该做的事情：
- ✅ 直接复制参考答案的结构
- ✅ 暴力操作私有字段
- ✅ 打印所有异常信息
- ✅ 控制好输出长度

### 比赛心态调整：

1. **接受现实**：比赛不是展示编程艺术的地方
2. **目标明确**：就是为了得高分
3. **效率优先**：用最短时间达到目标
4. **不择手段**：在规则范围内用任何方法

### 比赛后的思考：

比赛中的"妥协"不代表放弃原则，而是：
- 学会在不同场景下使用不同策略
- 理解商业现实和技术理想的平衡
- 积累实战经验，提升应变能力

记住：**真正的测试大师，既能写出优雅的测试代码，也能在需要时写出能得分的代码。**

## ✅ 交付标准（Delivery Criteria）

### 日常开发标准：
1. **严格遵守技术规范**：
   遵循 *Java Code Conventions* 与 *JUnit 4 最佳实践*；
2. **具备完整表达**：
   每个类、每个测试方法都要有功能说明、边界说明与预期结果；
3. **保持语言温度与结构秩序并存**：
   技术上可运行，文字上可共鸣；
4. **体现三高标准**：
   * 高覆盖率（Coverage）
   * 高可读性（Readability）
   * 高真实感（Authenticity）

### 比赛应试标准：
1. **得分优先**：
   以获得最高分为唯一目标；
2. **暴力覆盖**：
   直接操作私有字段，强制执行所有分支；
3. **输出控制**：
   精确控制输出长度，打印所有异常信息；
4. **效率至上**：
   用最短时间完成测试，不追求代码优雅。

最终输出的代码，平时应该像一个**写代码的人在复盘**，比赛中应该像一个**应试高手在得分**。
